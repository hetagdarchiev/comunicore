# Comunicore

> **Fullstack-платформа** для создания тематических сообществ и глубокого анализа социальных исследований/взаимодействий. Проект разработан в рамках курсовой работы с фокусом на архитектурные паттерны, автоматизацию кодогенерации и соответствие стандартам production-ready приложений.

---

## 📐 Архитектурный подход: Contract-First

В основе приложения лежит принцип **Single Source of Truth (Единый источник правды)**. 

Взаимодействие между клиентом и сервером жестко регламентировано спецификацией **OpenAPI 3.1.2** (`config/comunicore-api.yaml`). Из этого файла автоматически генерируется вся сетевая логика:
1. **На бэкенде:** Серверная часть, роутинг, структуры моделей и валидация входящих запросов (через `ogen`).
2. **На фронтенде:** API-клиент на базе нативного Fetch API с полной типизацией запросов и ответов (через `@hey-api/openapi-ts`).

---

## 🎯 Ключевой функционал

* **Управление сообществами:** Создание публичных/приватных форумов, гибкая модерация и ролевая модель участников.
* **Контентная система:** Посты с полной поддержкой Markdown, подсветкой синтаксиса кода и встроенной санитизацией на стороне клиента; древовидные вложенные комментарии.
* **Поведенческая экономика:** Рейтинговая система постов/комментариев и динамический расчет кармы пользователей.
* **Социальная аналитика:** Модуль для построения графов взаимодействий, анализа активности аудитории и связей внутри комьюнити.

## 🛠 Технологический стек и инфраструктура

### 💻 Frontend
Проект спроектирован по методологии **Feature-Sliced Design (FSD)**. Это обеспечивает слабую связанность модулей и разделение ответственности на слои: `app`, `widgets`, `features`, `entities`, `shared`.

| Категория | Стек / Технологии | Описание и назначение |
| :--- | :--- | :--- |
| **Core** | `Next.js 16.1.x` · `React 19` · `TS 5` | Современная компонентная база и строгая типизация. |
| **Сборка** | `Static Export` (`output: 'export'`) | Компиляция в чистую статику (`out/`) для максимальной производительности. |
| **Сетевой слой** | `TanStack Query v5` · `Fetch API` | Асинхронный стейт-менеджмент и кэширование без использования Axios. |
| **Формы** | `react-hook-form` · `Zod v4` | Валидация данных на клиенте по схемам, дублирующим OpenAPI. |
| **Стили и UI** | `Tailwind CSS v4` · `PostCSS` · `react-icons` | Современная утилитарная стилизация со сборкой через PostCSS. |
| **Контент** | `markdown-it` · `isomorphic-dompurify` | Безопасный рендеринг пользовательского Markdown-текста с защитой от XSS. |
| **Качество** | `ESLint 9` · `Prettier` · `Stylelint` · `Husky` | Автоматическая проверка кода перед коммитом (`lint-staged`). |

### ⚙️ Backend
Построен на чистом Go с минимальным использованием сторонних абстракций для сохранения контроля над производительностью, ресурсами и памятью.

| Категория | Стек / Технологии | Описание и назначение |
| :--- | :--- | :--- |
| **Core** | `Go 1.26` · `net/http` | Стандартная библиотека рантайма для HTTP-сервера без тяжелых фреймворков. |
| **API Generation** | `ogen` | Генерация серверной логики, валидации моделей и security-хендлеров из OpenAPI. |
| **Слой данных** | `PostgreSQL 18` · `pgx/v5` · `sqlc` | Компиляция сырого SQL-кода в типобезопасные и быстрые Go-методы. |
| **Миграции** | `golang-migrate` | Интегрировано в собственное Go-приложение, использующее общий TOML-конфиг сервера для подключения к БД (без внешних CLI-утилит). |
| **Безопасность** | `Argon2id` · `HttpOnly Cookie` | Сессионная модель авторизации (`sid`) без использования JWT. Хеширование паролей через Argon2id. |
| **Медиа-сервер** | `SHA-256 File Storage` | Сквозное хеширование загружаемых файлов для предотвращения дублирования данных. |
| **Автоматизация** | `just` (`backend/justfile`) | Использование task runner для сборки, линтинга и кодогенерации. |

### 🚀 CI/CD, Тестирование и DevOps

```
[ Локальная разработка ] ➔ [ GitHub Actions ] ➔ [ Анализ CodeQL ] ➔ [ Деплой через Tar/SSH ] ➔ [ Nginx / systemd (Ubuntu 26.04) ]
```

* **Тестирование:** Полное покрытие интеграционными тестами с использованием `testcontainers` (динамический подъем изолированной базы данных PostgreSQL в контейнере во время тестов) и сквозное E2E-тестирование API через `httpexpect`.
* **CI-конвейер:** В GitHub Actions автоматизирован билд фронтенда (на Node 22) и бэкенда (на Go 1.26 под Ubuntu), запуск тестов, а также глубокое сканирование безопасности через **CodeQL Advanced**.
* **Окружение (Production):** Прямое развертывание на выделенном сервере под управлением **Ubuntu 26.04 LTS** без использования Docker. Бэкенд-сервисы автоматизированы и управляются через **systemd**. Поскольку Go компилирует полностью автономные статические бинарники (без внешних зависимостей типа динамических библиотек), сборка с CI прозрачно и стабильно работает на целевой OS. В качестве веб-сервера и реверс-прокси используется **Nginx**, который раздает статический фронтенд из `/out` и перенаправляет запросы к Go-бинарнику.

## 📊 Структура хранения данных (БД)

1.  **Слой пользователей (`Users`):** Хранение учетных записей, авторизационных данных, метаинформации профилей и текущего баланса кармы.
2.  **Слой сообществ (`Communities`):** Управление сущностями форумов, конфигурацией приватности, правилами и ролевой моделью модераторов.
3.  **Слой контента (`Posts & Comments`):** Организация структуры публикаций и медиафайлов.
    > 📌 *Примечание архитектора:* Для выборки бесконечно вложенных цепочек комментариев без лишней нагрузки на СУБД в бэклог внесено внедрение рекурсивных запросов (**Recursive CTE**) на уровне SQL-миграций.
